2. CREATE THE CORE MULTI-TENANT DATA MODEL
==================================================

This application is organization-based.

A user may belong to MULTIPLE organizations and may have a DIFFERENT role in each organization.

DO NOT store one global application role on the user.

For example:

Jesse:
- Admin at Organization A
- Learner at Organization B

Create or prepare SQL migrations for these initial tables:

PROFILES

- id UUID primary key referencing auth.users(id)
- email
- first_name
- last_name
- avatar_url nullable
- created_at

ORGANIZATIONS

- id UUID primary key
- name
- slug unique
- logo_url nullable
- created_at

ORGANIZATION_MEMBERSHIPS

- id UUID primary key
- organization_id foreign key
- user_id foreign key
- role
- status
- created_at

Valid roles:

owner
admin
manager
learner

Valid membership status should include:

active
inactive
invited

Prevent duplicate memberships between the same user and organization.

Add appropriate foreign keys, constraints, and indexes.

Do not destructively modify existing Supabase tables without first identifying and explaining what already exists.

If existing tables can be reused or migrated safely, prefer that instead of unnecessarily duplicating them.

6. ROW LEVEL SECURITY
==================================================

Security is extremely important because this is a multi-tenant SaaS application.

Enable or prepare Row Level Security policies for exposed public tables.

The database must enforce organization isolation.

A user must never be able to access another organization's data simply by changing:

a URL
an organization ID
a request parameter
frontend state

Implement or generate RLS policies so that:

- Users can access their own appropriate profile information.
- Users can see organizations where they have active membership.
- Users can see their own memberships.
- Organization members can only access data allowed within organizations they belong to.
- Admin/owner/manager access is determined from organization_memberships.
- Learners cannot access organization administration data.
- Users cannot access data from organizations where they have no membership.

Do not rely only on frontend role checks.

Frontend authorization is for UX.

Database/backend authorization is for security.

Before executing potentially destructive SQL or RLS changes, explain what will change.
