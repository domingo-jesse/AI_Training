I am rebuilding my Training Simulator project from scratch. Please recreate it as a clean, production-ready web app while preserving the product behavior, workflows, and database concepts from my current Streamlit/Postgres prototype.

Project goal:
Build a role-based training simulator for organizations. Admins create and assign training modules; learners complete assigned scenario-based exercises; admins review submissions, use AI-assisted grading, track progress, and manage accounts. The app should be rebuilt cleanly with maintainable architecture, clear separation between UI, services, database access, authentication, and AI integrations.

Core technology requirements:
- Web app runnable on Replit.
- Use Supabase/Postgres as the primary database; do not rely on local SQLite for production behavior.
- Environment variables/secrets should drive configuration: DATABASE_URL or Supabase database URL, OPENAI_API_KEY, OPENAI_MODEL, Google OAuth credentials if enabled.
- Keep the app secure enough for real users: hashed passwords with a modern algorithm, parameterized SQL/ORM queries, role-based access control, no secrets in code, safe error handling, audit logs, and organization-level data isolation.
- Preserve or improve the existing app's responsive UI, admin dashboards, learner workspace, and grading workflows.

Roles and permissions:
- Admin users can manage modules, assignments, learners, grading, progress reports, accounts, database/debug tools when allowed, QA/testing tools, profile, and settings.
- Learner users can see their home page, assigned modules, module workspace, progress/results, profile, and settings.
- Developer/test accounts may see extra tools such as database tables, debug logs, and QA Test Center; regular users should not.
- Every user belongs to an organization. All module, assignment, attempt, grading, and learner data must be scoped by organization.

Authentication and account features:
- Login page with email/username and password.
- Account creation flow.
- Optional Google OAuth support.
- Profile page and settings page.
- User records should include name, email, role, team, organization, auth provider, active/inactive status, username/password hash when applicable, and created/updated timestamps.

Main admin features to rebuild:
1. Admin Dashboard
   - Overview metrics for learners, modules, assignments, submissions, completion status, and recent activity.
   - Clear navigation to major admin areas.

2. Assignment Management
   - Create assignments by selecting modules and learners.
   - Support due dates, active/inactive assignments, bulk assignment workflows, and assignment tracking records.
   - Current assignments table with learner, team, module, due date, assigned date, active state, attempt count, last attempt, and status: Not Started, In Progress, Completed, or Overdue.
   - Ability to filter learners/modules and avoid assigning archived modules.

3. Submission Grading / Grading Center
   - List submitted attempts needing review.
   - Show learner responses, question-level scores, AI feedback, manual feedback, expected answers, grading criteria, and overall scores.
   - Support grading statuses such as submitted, pending review, AI grading, AI graded pending review, approved, returned, and grading failed.
   - Allow admins to approve grades, adjust scores, add feedback, regrade submissions, and control what learners can see.
   - Store overall submission scores and question-level scores.
   - Keep regrade history.

4. Progress Tracking
   - Track learners, teams, modules, completion rates, statuses, last activity, attempts, and scores.
   - Provide filters and summary metrics.
   - Include learner profiles, module assignments, and module progress records.

5. Account Management
   - Create admin and learner accounts.
   - Edit users, teams, roles, status, and organization.
   - Distinguish active and inactive learners.

6. Module Builder
   - Build modules manually and with AI assistance.

   - Questions should support:
     - open_text
     - multiple_choice
     - ai_conversation
   - Question fields should include order, question text, rationale, question type, options/choices, expected answer or answer guidance, scoring method, max points, keyword expected terms, LLM grading criteria, learner-visible feedback mode, AI conversation prompt, AI role/persona, evaluation focus, max learner responses, optional wrap-up instruction, and source generation run.
   - Support module-level rubric criteria with criterion name, description, max points, and display order.
   - Save modules as draft/existing/published/archived as appropriate.

7. Manage Modules
   - View module list with status and metadata.
   - Edit modules and questions.
   - Archive/unarchive or manage availability.
   - Preserve generated module/question content.

8. Database Tables / Debug Logs / QA Test Center
   - Developer-only tools for inspecting public database tables, recent logs, and running QA checks.
   - Do not expose these to regular admins or learners unless explicitly configured.
